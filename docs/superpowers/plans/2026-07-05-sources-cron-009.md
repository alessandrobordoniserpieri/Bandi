## Architecture decision — app ↔ scraper boundary (refines the recap's "no import" rule)
ADR-002's real constraint is **the scraper never imports `app/`**. The reverse is allowed:
the app's cron route consumes the scraper as a library. The scraper exposes a single public
entrypoint `scraper/src/index.ts` (`runProductionScrape`, `runScrape`, key types). The app
depends on it via a `file:../scraper` dependency named `bandi-scraper` and lists it in
`transpilePackages` (Next 16 App Router bundles a raw-TS workspace dep into the route
handler — confirmed in the bundled docs). The scraper stays a self-contained package with no
knowledge of the app. Recorded as ADR-008.

## Files (scraper)
1. `pipeline/types.ts` — extend `SourceConfig` with optional `scrapeConfig?: ScrapeConfig`
   (`{ listUrl?; maxPages?; waitFor? }`). Additive; existing pipeline/tests unaffected.
2. `pipeline/browserless-fetcher.ts` — `BrowserlessFetcher implements PageFetcher`. POSTs to
   Browserless `/content` (`BROWSERLESS_URL` default `https://chrome.browserless.io`, token
   `BROWSERLESS_API_KEY`), 30s timeout, 1 retry via the shared `withRetry` (retryable
   `ProviderError`). Returns one `RawPage` for the list url (pagination deferred). Injectable
   `fetchImpl`.
3. `db/supabase-grants-db.ts` — `SupabaseGrantsDb implements GrantsDb` over a
   `@supabase/supabase-js` service-role client. Pure, exported mappers do the real work and
   are unit-tested: `grantToInsertRow` (camel→snake, **status null → 'aperto'** to respect the
   not-null column), `patchToUpdateRow`, `rowToStoredGrant`, `rowToSourceConfig`.
4. `sources/index.ts` — `loadEnabledSources(client)`: selects `grant_sources` where
   `enabled = true`, maps rows → `SourceConfig` (incl. `scrapeConfig`).
5. `run-production.ts` — `runProductionScrape(env)`: builds `getProvider(env)`,
   `BrowserlessFetcher`, `SupabaseGrantsDb`, loads enabled sources (optionally filtered by
   slug), calls `runPipeline`, returns `PipelineResult[]`.
6. `cli.ts` — `npm run scrape [--source=<name>] [--dry-run]` for manual/debug runs
   (`--dry-run` skips DB writes via a logging no-op `GrantsDb`).
7. `index.ts` — public API re-exports.
8. `package.json` — name `bandi-scraper`, `exports`/`main` → `src/index.ts`, add
   `@supabase/supabase-js` dep + `tsx` dev-dep + `"scrape"` script.
9. `env.example` — add `BROWSERLESS_API_KEY`, `BROWSERLESS_URL`.

## Files (app)
10. `src/app/api/cron/scrape/route.ts` — `POST`, Node runtime, `maxDuration`. Rejects with
    `401` unless `Authorization: Bearer ${CRON_SECRET}` matches (constant-time compare via a
    small `isAuthorized` helper). On success imports `runProductionScrape` from
    `bandi-scraper`, runs it, logs and returns the `PipelineResult[]` summary.
11. `src/app/api/cron/auth.ts` — pure `isAuthorized(header, secret)` (testable).
12. `next.config.ts` — `transpilePackages: ["bandi-scraper"]`.
13. `package.json` — add `"bandi-scraper": "file:../scraper"`.
14. `vercel.json` — `{ "crons": [{ "path": "/api/cron/scrape", "schedule": "0 3 */2 * *" }] }`.
15. `supabase/migrations/0005_enable_sources.sql` — `enabled = true` on the 12 MVP sources +
    per-source `scrape_config` refinement.

## Acceptance (roadmap) & how verified
- Manual run on a real source inserts valid grants / no dupes on rerun / unreachable source
  records `last_error` without blocking others → **manual, documented in the PR** (needs real
  `BROWSERLESS_API_KEY` + service role; the dedup/isolation logic itself is unit-tested here
  and in 007).
- Cron visible/active in Vercel → after deploy (manual).
- Offline: scraper + app suites green, both `tsc` clean, `next build` succeeds with the
  transpiled scraper dep.
