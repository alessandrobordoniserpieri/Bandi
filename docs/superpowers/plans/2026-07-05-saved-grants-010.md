# Plan — branch 010 `feat/010-saved-grants`

> Depends on 005 (grant display). The "I miei bandi" page: a personal 4-state pipeline + 2
> outcomes (§5.2) as a kanban, per-grant notes, and the virtuous loop — moving a grant to
> **Finanziato** auto-appends a track-record row to profile §7 (I6).

## Source of truth
- Roadmap `#b010`; §5.2 (states) and §7 (track record).
- Schema: `saved_grants` (unique(user_id, grant_id), status enum, notes, `track_record_written`),
  `profiles.project_history jsonb` — rows stored **snake_case**
  `{grant_name, provider_id, year, outcome, amount, kind}` (see `parseProjectHistory`).
- RLS: owner-only select/insert/update/delete on `saved_grants` and `profiles`.

## Design decisions
- **Atomic mark-funded → Postgres RPC.** The roadmap requires "both writes or neither", which
  is a real transaction across `saved_grants` and `profiles`. A `security invoker` function
  `set_saved_grant_status(p_saved_grant_id, p_status)` does it in one transaction (RLS applies
  as the caller). On `finanziato` when `track_record_written = false` it appends the row from
  the grant and sets the flag. Migration `0006`. Verified against the real DB via MCP.
- **State machine in TS** (`status.ts`) drives the UI (valid transitions only) and is
  unit-tested; the action re-validates the transition server-side before the RPC.
- Client components call the `"use server"` actions directly (no drag&drop — not required).

## Tests (offline)
- `status.test.ts` — valid/invalid transitions both directions; `nextStatuses`; meta present
  for every status.
- `actions.test.ts` (mocked `@/lib/supabase/server`) — `saveGrant` inserts (user+grant);
  `removeSavedGrant` deletes by user+grant; `updateNotes` updates; `updateStatus` rejects an
  invalid transition and calls `rpc('set_saved_grant_status', …)` on a valid one.
- Real-DB integration (via MCP, documented in PR — not in the vitest suite): funded appends the
  right row + sets the flag; a second funded does **not** add a second row; double `saveGrant`
  → one row (unique). This is stronger than a mock for the field mapping.

## Acceptance (roadmap)

- Finanziato → profile §7 shows the new row with no data entry; track-record dimension (6)
  improves on refresh.
- Notes persist per grant, owner-only (RLS).
